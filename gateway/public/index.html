Persistent Memory Engine for AI Agents

AtlasRAG provides durable memory APIs, multi-tenant governance, and retrieval workflows so teams can store, recall, and operationalize knowledge for agents in production. It combines a fast C++ vector store with a secure gateway that orchestrates embeddings, recall, and background jobs.

AtlasRAG Product Overview

AtlasRAG is a persistent memory engine for AI agents. It combines a fast vector index with structured memory APIs, multi-tenant governance, and developer-grade observability so teams can ship reliable, long-lived agents.

Memory API

Write durable memories with clear types (artifact, semantic, procedural, episodic, conversation, summary). Each memory is attributed to a tenant, collection, and principal for safe isolation.

Recall API

Query memories by vector similarity plus filters like type, time range, and collection. Recall respects visibility and ACLs for enterprise-grade access control.

Reflection & Summarization

Convert artifacts into semantic/procedural/summary memories using asynchronous jobs, with provenance links back to original sources.

Collections

Segment knowledge by product, team, or environment. Collections scope indexing, recall, and governance without creating separate stacks.

Multi-tenant Architecture

Tenant IDs are enforced at the API layer and in storage namespaces. Every vector and memory item is isolated and query-safe by default.

Governance

Fine-grained ACLs and visibility modes (tenant, private, ACL) keep sensitive knowledge constrained to the right users or service principals.

Metrics

Track ingestion throughput, vector-store performance, and gateway latency percentiles per tenant. Developers can spot hotspots before they impact production traffic.

Reliability Features

Idempotency keys, TTL cleanup, and compaction jobs reduce operational risk and keep memory stores healthy over time.

Developer Ready

OpenAPI docs, Swagger UI, and a Node SDK make it easy to integrate AtlasRAG into agents, applications, or internal platforms.

Typical Flow

  1. Index or upload artifacts into a collection.
  2. Trigger reflection jobs to produce durable memories.
  3. Recall memories with filters and ACL-aware access.
  4. Monitor metrics for latency, throughput, and quality.

Built for Teams

AtlasRAG supports multi-tenant workflows, SSO-ready authentication, and service tokens for automation. Organizations can onboard new teams without standing up separate stacks.

Authentication

Protected endpoints require either a JWT or a service token. Send Authorization: Bearer <jwt> or X-API-Key: <token> (also supports Authorization: ApiKey <token>).

Multi-tenant isolation is enforced by the auth token. JWTs must include a tenant identifier using one of these claims: tenant, tid, or sub. Service tokens inherit the tenant from the admin who created them.

Admins can issue service tokens via POST /v1/admin/service-tokens. Store the returned token securely; it is shown only once.

Create API key (admin)
curl -X POST http://localhost:3000/v1/admin/service-tokens \
  -H "Authorization: Bearer YOUR_ADMIN_JWT" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "ci-pipeline",
    "principalId": "svc:ci",
    "roles": ["indexer"],
    "expiresAt": "2026-12-31T00:00:00Z"
  }'

Requires role admin (or a token whose principal matches the tenant).

Tenant auth mode

Admins can set per-tenant login policy: sso_only, sso_plus_password, or password_only. sso_only disables password login; password_only disables SSO.

You can also restrict which SSO providers are allowed by supplying ssoProviders (google, azure, okta).

Get auth mode (admin)
curl -X GET http://localhost:3000/v1/admin/tenant \
  -H "Authorization: Bearer YOUR_ADMIN_JWT"
Set auth mode (admin)
curl -X PATCH http://localhost:3000/v1/admin/tenant \
  -H "Authorization: Bearer YOUR_ADMIN_JWT" \
  -H "Content-Type: application/json" \
  -d '{ "authMode": "sso_only", "ssoProviders": ["google"] }'

Valid values: sso_only, sso_plus_password, password_only.

Security
Built-in controls
  • Auth required: JWTs or service tokens protect all sensitive endpoints.
  • Tenant isolation: data is partitioned by tenant and enforced in every query.
  • Role-based access: reader/indexer/admin control access to privileged actions.
  • Tenant auth mode: enforce sso_only or password_only per tenant.
  • Tenant SSO allowlist: restrict SSO to google, azure, or okta.
  • Visibility + ACL: tenant, private, acl rules apply to search/recall.
  • Idempotency: write/index/reflect accept Idempotency-Key to prevent double writes.
  • Audit logs: sensitive actions (auth policy changes, key revokes, deletes) are recorded with actor + request metadata.
  • Rate limits + lockout: login throttling + failed-login lockout are enabled.
  • URL ingestion safety: private IP ranges are blocked (SSRF protection).
  • Prompt injection guard: source sanitization is enabled by default.

Architecture
  • Gateway (Node) handles auth, routing, and orchestration.
  • Vector store (C++ TCP service) stores embeddings and serves similarity search.
  • Postgres stores chunks, memory items, links, jobs, and idempotency keys.
  • OpenAI (or compatible) provides embeddings and generation.
  • Background jobs handle reflection, summarization, redundancy scoring, and lifecycle tasks.
  • Expired items are automatically swept and removed (vectors + DB rows), preventing orphan vectors.
  • Jobs retry with exponential backoff before transitioning to failed.
  • Job reruns are idempotent (derived memories are replaced, not duplicated).
  • Structured logs include request_id, tenant_id, and collection.
  • Prometheus metrics are exposed at /metrics (scoped to the tenant; admin sees all tenants).

Technical White Paper

Adaptive Memory Value + Lifecycle (AMV-L)

A scoring and control policy for maintaining high-signal, bounded long-term memory in AtlasRAG.

Objective
Optimize retention quality under cost and safety constraints.
Output
Per-item action: retain, delete, compact, or promote.
Execution Model
Event-driven updates plus periodic sweeps.
Outline
Overview Problem framing Value function Reuse (R) Utility (U) Novelty (N) Trust (T) Recency (A) Importance hint (H) Cost (C) Lifecycle policy Delete operator Compaction operator Promotion operator TTL precedence Signals and events System properties Conclusion
1. Overview

AtlasRAG implements Adaptive Memory Value + Lifecycle (AMV-L), a control algorithm that continuously estimates memory utility and applies lifecycle actions to each item. AMV-L computes a bounded value score from observed behavior and then maps the score to deterministic lifecycle decisions.

The system-level goal is to preserve memories that improve downstream outcomes while limiting storage growth, retrieval noise, and token cost.

2. Problem Framing

Naive retention strategies are insufficient in production memory systems:

  • Store-everything policies produce unbounded growth and retrieval dilution.
  • Fixed TTL-only policies remove stale data but may discard high-value knowledge.

These failure modes increase operational cost and reduce answer quality. AMV-L reframes retention as an optimization problem over value, trust, novelty, and cost.

3. Value Function V(m)

Each memory item m is assigned V(m) in [0, 1] via a weighted logistic model:

Equation (Implementation-Accurate)
V = sigmoid(
  1.2 * R +
  2.0 * U +
  1.0 * N +
  1.0 * T +
  0.5 * A +
  1.0 * H -
  1.0 * C
)
Signal Definitions and Transforms
Signal Source Field(s) Transform / Range
R reuse_count log1p(reuse_count) (unbounded, sublinear)
U utility_ema Clamped to [-1, 1]
N redundancy_score 1 - redundancy_score, redundancy clamped to [0, 1]
T trust_score Clamped to [0, 1], default 0.5
A last_used_at / created_at Exponential time decay in [0, 1]
H importance_hint Clamped to [-1, 1], default 0
C tokens_est tokens_est / cost_scale, clamped to [0, 2] (default scale 2000)

Final value is sigmoid-normalized and clamped to [0, 1].

Notation
Symbol Interpretation
RReuse intensity
UOutcome utility (EMA)
NNovelty (= 1 - redundancy)
TTrust estimate
AAge/recency contribution
HUser-provided importance prior
CCost penalty
4. Reuse Signal (R)

Reuse captures empirical demand for a memory item through repeated retrieval. AMV-L uses R = log1p(reuse_count), which dampens late-stage growth and prevents very old high-frequency items from dominating value indefinitely.

5. Utility Signal (U)

Utility estimates whether a memory improves downstream task outcomes. Utility is updated as an exponential moving average from runtime events and clamped to [-1, 1].

  • Positive contributors: successful use in answers, positive user/task outcomes.
  • Negative contributors: explicit negative feedback and failed task outcomes.
6. Novelty Signal (N)

Novelty is derived from semantic redundancy. Redundancy is estimated via nearest-neighbor similarity; novelty is computed as N = 1 - redundancy.

Redundancy scoring runs asynchronously for newly written/updated items, with periodic sweeps to ensure eventual consistency across older items.

7. Trust Signal (T)

Trust captures reliability of the memory source and subsequent validation outcomes. In the current implementation, trust starts at 0.5 and moves with feedback/task events.

Additional source-quality priors can be layered on top without changing the lifecycle policy interface.

8. Recency Signal (A)

Recency applies a temporary boost to recently used memories through exponential decay. This improves responsiveness to changing context while naturally fading stale items over time.

9. Importance Prior (H)

Importance is an explicit user/system prior in [-1, 1] (default 0). It allows intentional bias toward business-critical memories and can be combined with pinning when strict retention is required.

10. Cost Penalty (C)

Cost approximates retrieval and storage overhead using estimated token size and scales as tokens_est / cost_scale, clamped to [0, 2]. Default cost scale is 2000.

11. Lifecycle Policy

AMV-L maps value to lifecycle operators using thresholded policy rules:

12. Delete Operator

Delete performs hard removal from metadata storage and vector storage. It is selected for expired items, low-value items, or policy-driven cleanup decisions.

13. Compaction Operator

Compaction summarizes multiple related low-value memories into a denser representation. Original items may be archived or removed after summary write to reduce noise and token cost.

14. Promotion Operator

Promotion distills high-value episodic/artifact context into reusable higher-order knowledge (for example semantic facts or procedural guidance), improving retrieval precision in future tasks.

15. TTL Precedence

TTL enforces a hard compliance retention boundary. AMV-L operates only on non-expired memory, and TTL always has higher precedence than value-based retention.

16. Signals and Event Ingestion

AtlasRAG ingests runtime signals automatically and accepts explicit supervisory feedback:

  • retrieved and used_in_answer during recall/answer flows.
  • user_positive and user_negative via POST /v1/feedback.
  • task_success and task_fail via POST /v1/memory/event.

Event ingestion triggers metric updates and value recomputation, and is reinforced by periodic decay sweeps.

17. Resulting System Properties
  • Bounded long-term memory growth.
  • Higher retrieval signal-to-noise ratio.
  • Lower token and storage overhead.
  • Continuous adaptation to real usage outcomes.
18. Conclusion

AMV-L provides AtlasRAG with a self-optimizing memory substrate that explicitly balances value, trust, novelty, and cost. The result is a practical lifecycle policy that learns what to retain and what to compress or forget based on observed behavior, rather than static retention rules alone.
