Gateway documentation

Technical usage guide for the AtlasRAG semantic gateway, including setup, API usage, and operations.

Overview

AtlasRAG lets you index documents, run semantic search, and ask questions across your knowledge base. Search and Ask return the most relevant chunks and provide citations.

Quick start (users)

  1. Open the AtlasRAG web app in your browser.
  2. Sign in using your organization’s login method.
  3. Index a document, then use Search or Ask.

Authentication

Protected endpoints require a Bearer token in the Authorization header. The UI stores the token in localStorage only.

Multi-tenant isolation is enforced by the JWT. Include a tenant identifier in the token using one of these claims: tenant, tid, or sub. All data is namespaced by that value.

If your organization uses SSO (Google, Microsoft, Okta), choose the SSO option on the Settings page. Admins control which providers are enabled.

POST /login
{
  "username": "admin",
  "password": "changeme"
}
curl -H "Authorization: Bearer YOUR_TOKEN" http://localhost:3000/stats

Indexing workflow (users)

Indexing splits text into chunks, embeds each chunk, stores vectors in the C++ TCP service, and stores chunk text in Postgres. Doc IDs must use only letters, numbers, dot, dash, or underscore.

  • Use POST /docs to index raw text.
  • Use POST /docs/url to fetch and index a URL.
  • Use GET /docs to list your indexed docs.
  • Use DELETE /docs/:docId to remove stored chunk text.
POST /docs
{
  "docId": "swe_notes",
  "text": "Your document text..."
}
POST /docs/url
{
  "docId": "swe_docs",
  "url": "https://example.com/article"
}
GET /docs
DELETE /docs/swe_docs

Note: delete removes stored chunk text; vector deletion is a future improvement.

The docs list is tenant-scoped and returns doc IDs with chunk counts. The UI uses this list to populate doc filters for Search and Ask.

Search

Search embeds your query and returns the Top K most similar chunks by cosine similarity. Higher K increases recall but can reduce precision and add latency.

Optionally pass docIds to restrict search to specific docs.

GET /search?q=write-ahead%20logging&k=5
GET /search?q=write-ahead%20logging&k=5&docIds=onboarding,design-notes

Ask (RAG)

Ask retrieves the Top K chunks, then generates a response using only those sources. The response is concise and avoids speculation.

Provide docIds to answer using a subset of docs, or omit to use all.

Source text is sanitized to reduce prompt injection risk. Set PROMPT_INJECTION_GUARD=0 to disable.

POST /ask
{
  "question": "Explain WAL and why it matters",
  "k": 5,
  "docIds": ["onboarding", "design-notes"]
}

Metrics (developers)

Metrics show vector-store stats plus gateway latency percentiles (p50/p90/p95/p99) for the most recent samples.

Limits

  • Document size cap: 200,000 chars (server truncates).
  • URL fetch cap: 1,000,000 chars (server truncates).
  • Client file cap: 5 MB.
  • Embeddings are batched; chunk size is capped for embedding safety.

Developer notes

  • Use concise doc IDs; avoid spaces to prevent TCP parsing errors.
  • Lower Top K if responses are slow or noisy.
  • Monitor Metrics for vector and command throughput.

Troubleshooting

  • 401 Unauthorized: ensure a valid JWT is included in Authorization.
  • Timeouts during indexing: try smaller documents or lower Top K on retrieval.
  • Context length errors: re-index with smaller chunks or lower Top K.
