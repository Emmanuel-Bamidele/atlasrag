Persistent Memory Engine for AI Agents

AtlasRAG provides durable memory APIs, multi-tenant governance, and retrieval workflows so teams can store, recall, and operationalize knowledge for agents in production. It combines a fast C++ vector store with a secure gateway that orchestrates embeddings, recall, and background jobs.

AtlasRAG Product Overview

AtlasRAG is a persistent memory engine for AI agents. It combines a fast vector index with structured memory APIs, multi-tenant governance, and developer-grade observability so teams can ship reliable, long-lived agents.

Memory API

Write durable memories with clear types (artifact, semantic, procedural, episodic, conversation, summary). Each memory is attributed to a tenant, collection, and principal for safe isolation.

Recall API

Query memories by vector similarity plus filters like type, time range, and collection. Recall respects visibility and ACLs for enterprise-grade access control.

Reflection & Summarization

Convert artifacts into semantic/procedural/summary memories using asynchronous jobs, with provenance links back to original sources.

Collections

Segment knowledge by product, team, or environment. Collections scope indexing, recall, and governance without creating separate deployments.

Multi-tenant Architecture

Tenant IDs are enforced at the API layer and in storage namespaces. Every vector and memory item is isolated and query-safe by default.

Governance

Fine-grained ACLs and visibility modes (tenant, private, ACL) keep sensitive knowledge constrained to the right users or service principals.

Metrics

Track ingestion throughput, vector-store performance, and gateway latency percentiles per tenant. Developers can spot hotspots before they impact production traffic.

Reliability Features

Idempotency keys, TTL cleanup, and compaction jobs reduce operational risk and keep memory stores healthy over time.

Developer Ready

OpenAPI docs, Swagger UI, and a Node SDK make it easy to integrate AtlasRAG into agents, applications, or internal platforms.

Typical Flow

  1. Index or upload artifacts into a collection.
  2. Trigger reflection jobs to produce durable memories.
  3. Recall memories with filters and ACL-aware access.
  4. Monitor metrics for latency, throughput, and quality.

Built for Teams

AtlasRAG supports multi-tenant workflows, SSO-ready authentication, and service tokens for automation. Organizations can onboard new teams without standing up separate stacks.

Authentication

Protected endpoints require either a JWT or a service token. Send Authorization: Bearer <jwt> or X-API-Key: <token> (also supports Authorization: ApiKey <token>).

Multi-tenant isolation is enforced by the auth token. JWTs must include a tenant identifier using one of these claims: tenant, tid, or sub. Service tokens inherit the tenant from the admin who created them.

Admins can issue service tokens via POST /v1/admin/service-tokens. Store the returned token securely; it is shown only once.

Create API key (admin)
curl -X POST http://localhost:3000/v1/admin/service-tokens \
  -H "Authorization: Bearer YOUR_ADMIN_JWT" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "ci-pipeline",
    "principalId": "svc:ci",
    "roles": ["indexer"],
    "expiresAt": "2026-12-31T00:00:00Z"
  }'

Requires role admin (or a token whose principal matches the tenant).

Tenant auth mode

Admins can set per-tenant login policy: sso_only, sso_plus_password, or password_only. sso_only disables password login; password_only disables SSO.

You can also restrict which SSO providers are allowed by supplying ssoProviders (google, azure, okta).

Get auth mode (admin)
curl -X GET http://localhost:3000/v1/admin/tenant \
  -H "Authorization: Bearer YOUR_ADMIN_JWT"
Set auth mode (admin)
curl -X PATCH http://localhost:3000/v1/admin/tenant \
  -H "Authorization: Bearer YOUR_ADMIN_JWT" \
  -H "Content-Type: application/json" \
  -d '{ "authMode": "sso_only", "ssoProviders": ["google"] }'

Valid values: sso_only, sso_plus_password, password_only.

Security
Built-in controls
  • Auth required: JWTs or service tokens protect all sensitive endpoints.
  • Tenant isolation: data is partitioned by tenant and enforced in every query.
  • Role-based access: reader/indexer/admin control access to privileged actions.
  • Tenant auth mode: enforce sso_only or password_only per tenant.
  • Tenant SSO allowlist: restrict SSO to google, azure, or okta.
  • Visibility + ACL: tenant, private, acl rules apply to search/recall.
  • Idempotency: write/index/reflect accept Idempotency-Key to prevent double writes.
  • Audit logs: sensitive actions (auth policy changes, key revokes, deletes) are recorded with actor + request metadata.
  • Rate limits + lockout: login throttling + failed-login lockout are enabled.
  • URL ingestion safety: private IP ranges are blocked (SSRF protection).
  • Prompt injection guard: source sanitization is enabled by default.

Architecture
  • Gateway (Node) handles auth, routing, and orchestration.
  • Vector store (C++ TCP service) stores embeddings and serves similarity search.
  • Postgres stores chunks, memory items, links, jobs, and idempotency keys.
  • OpenAI (or compatible) provides embeddings and generation.
  • Background jobs handle reflection, summarization, redundancy scoring, and lifecycle tasks.
  • Expired items are automatically swept and removed (vectors + DB rows), preventing orphan vectors.
  • Jobs retry with exponential backoff before transitioning to failed.
  • Job reruns are idempotent (derived memories are replaced, not duplicated).
  • Structured logs include request_id, tenant_id, and collection.
  • Prometheus metrics are exposed at /metrics (scoped to the tenant; admin sees all tenants).
Deployment
  • Run with Docker Compose: docker compose up -d
  • Set required env: OPENAI_API_KEY, JWT_SECRET, Postgres credentials.
  • Timeouts: TCP_TIMEOUT_MS, DB_CONNECT_TIMEOUT_MS, DB_QUERY_TIMEOUT_MS, DB_STATEMENT_TIMEOUT_MS, OPENAI_TIMEOUT_MS.
  • Rate limiting: per IP (RATE_LIMIT_WINDOW_MS/RATE_LIMIT_MAX) and per tenant/key (TENANT_RATE_LIMIT_WINDOW_MS/TENANT_RATE_LIMIT_MAX).
  • Expose gateway on :3000 and front with HTTPS in production.
  • Optionally set PUBLIC_BASE_URL / OPENAPI_BASE_URL for correct docs links.

Adaptive Memory Value + Lifecycle (AMV-L)

AtlasRAG implements an adaptive memory lifecycle algorithm called Adaptive Memory Value + Lifecycle (AMV-L). AMV-L continuously estimates the value of each memory item based on observed usage and outcome signals, and automatically decides whether to retain, compact, promote, or delete memory.

This allows AtlasRAG to maintain bounded, high-signal long-term memory for agents.

Why AMV-L Exists

Naive memory systems:

  • Store everything forever
  • Or delete using fixed TTL

Both approaches lead to:

  • Memory bloat
  • Noisy retrieval
  • High token costs
  • Loss of important knowledge

AMV-L treats memory as an optimization problem rather than a storage problem.

Memory Value Score

Each memory item m has a value score: V(m) in [0,1].

Formula
V = sigmoid(
  1.2 * R +
  2.0 * U +
  1.0 * N +
  1.0 * T +
  0.5 * A +
  1.0 * H -
  1.0 * C
)
Ranges & Transforms (Code-Accurate)
Signal Source Range / Transform
R reuse_count R = log1p(reuse_count) (unbounded, sublinear)
U utility_ema Clamped to [-1, 1] (EMA target values in that range)
N redundancy_score N = 1 - redundancy_score, where redundancy is clamped to [0, 1]
T trust_score Clamped to [0, 1]; defaults to 0.5, updated by feedback/task events
A last_used_at / created_at Exponential decay in [0, 1] with a configurable half-life
H importance_hint Clamped to [-1, 1] (optional; default 0)
C tokens_est C = tokens_est / cost_scale, clamped to [0, 2] (default scale 2000)

V is computed with a sigmoid and then clamped to [0, 1].

Reuse (R)

Measures how often and how recently a memory is retrieved.

  • Rolling reuse count
  • Time since last used

In code, R = log1p(reuse_count), so reuse grows sublinearly and is not normalized to [0, 1].

Utility (U)

Measures whether memory actually improves outcomes.

Updated from events:

  • Memory used in answer
  • Task success/failure
  • User feedback

Utility is stored as an exponential moving average and clamped to [-1, 1].

Novelty (N)

Measures redundancy relative to other memories.

Computed using nearest-neighbor similarity:

  • High similarity - high redundancy
  • Novelty = 1 - redundancy

Redundancy is computed asynchronously for new or updated memory items, with a nightly sweep as a fallback.

Trust (T)

Represents source reliability. In the current implementation:

  • Defaults to 0.5
  • Adjusted by feedback and task events (positive increases, negative decreases)

Source-derived trust heuristics can be layered on later, but are not applied automatically today.

Recency (A)

Recent memories get a small temporary boost.

Decays exponentially over time.

Importance Hint (H)

Optional user-provided hint in [-1, 1] (default 0).

Used for pinning critical memories.

Cost (C)

Estimates storage and retrieval cost based on:

  • Token size
  • Chunk size

In code, C = tokens_est / cost_scale and is clamped to [0, 2] (default scale 2000).

Lifecycle Decisions

AMV-L uses thresholds:

Condition Action
expireddelete
pinnedkeep
V < 0.25delete
0.25 <= V < 0.45compact
V > 0.70promote
otherwiseretain
Delete

Memory is permanently removed from the metadata store and the vector engine. This is used for low-value or expired memory.

Compact (Summarize)

Multiple low-value related memories are summarized into a single memory item. Original items may be archived or deleted after the summary is created.

Promote (Distill)

High-value memory is distilled into higher-level memory types such as semantic facts and procedural steps. This converts raw experience into reusable knowledge.

Hard TTL vs AMV-L

TTL is a hard retention boundary for compliance.

AMV-L operates only on non-expired memory.

TTL always wins.

Signals and Events

AtlasRAG records signals automatically and exposes explicit feedback:

  • retrieved and used_in_answer are logged during recall and answers
  • user_positive and user_negative are submitted via POST /v1/feedback
  • task_success and task_fail can be logged via POST /v1/memory/event

Utility is updated as an exponential moving average, and value scores are recomputed after events and on periodic decay sweeps.

Benefits
  • Bounded memory growth
  • Higher retrieval signal-to-noise
  • Lower token usage
  • Continuous improvement
Summary
