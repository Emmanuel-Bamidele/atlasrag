Supavector API Documentation

This page is for developers building apps and agents. The simple model is: get a service token once, store SUPAVECTOR_BASE_URL and SUPAVECTOR_API_KEY, then call SupaVector from your backend, worker, or agent runtime.

Recommended Flow

  1. If you self-host, run supavector onboard once. If SupaVector already exists, ask the operator for a base URL and service token.
  2. If you are in Python, run python3 -m pip install supavector.
  3. Store SUPAVECTOR_BASE_URL and SUPAVECTOR_API_KEY in your runtime env.
  4. Index with /v1/docs, then retrieve with /v1/search, /v1/ask, /v1/code, /v1/boolean_ask, or memory endpoints.

Usage Modes

  • Self-host with the bundled stack when you want to run SupaVector yourself.
  • Use your own Postgres when you already have database and secret-management infrastructure.
  • Use a shared or hosted deployment when your app just needs a running API and a token.
  • Keep your own product auth in front when SupaVector should sit behind your backend.

Important Notes

  • JWTs are for human admin sessions. Service tokens are for apps, agents, workers, and CI.
  • If you want SupaVector to keep its own auth/runtime but use your provider key for a request, send the matching request-scoped provider-key header.
  • Idempotency-Key is required for /v1/docs, /v1/docs/url, /v1/memory/write, /v1/memory/reflect.
  • /v1/docs/url only indexes reachable public URLs. Private/local hosts are blocked.
  • Use /v1/jobs/{id} until the job status becomes succeeded or failed.
  • /v1/boolean_ask returns only true, false, or invalid, plus supportingChunks.
Self-host once or get a token from an existing deployment, then reuse that base URL and service token everywhere else.
